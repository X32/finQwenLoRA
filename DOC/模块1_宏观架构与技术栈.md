### 4. 灵活的数据加载策略

**问题**：不同数据集大小需要不同的处理策略

**解决方案** (第209-264行)：

#### SupervisedDataset (第209-231行)

```python
class SupervisedDataset(Dataset):
    """预处理模式：适合中小数据集"""
    def __init__(self, raw_data, tokenizer, max_len, system_message):
        # 启动时处理所有数据
        sources = [example["conversations"] for example in raw_data]
        data_dict = preprocess(sources, tokenizer, max_len, system_message)
      
        # 存储处理后的数据
        self.input_ids = data_dict["input_ids"]
        self.labels = data_dict["labels"]
        self.attention_mask = data_dict["attention_mask"]
```

**优势**：

- ✅ 训练时速度快（无需实时处理）
- ✅ 数据一致性高
- ✅ 调试简单

**劣势**：

- ❌ 启动慢（需要处理所有数据）
- ❌ 内存占用大（存储所有处理后的数据）

#### LazySupervisedDataset (第234-263行)

```python
class LazySupervisedDataset(Dataset):
    """懒加载模式：适合大型数据集"""
    def __getitem__(self, i) -> Dict[str, torch.Tensor]:
        # 首次访问时检查缓存
        if i in self.cached_data_dict:
            return self.cached_data_dict[i]
      
        # 首次访问时才处理
        ret = preprocess([self.raw_data[i]["conversations"]], ...)
      
        # 缓存结果避免重复计算
        self.cached_data_dict[i] = ret
        return ret
```

**优势**：

- ✅ 启动速度快（跳过数据预处理）
- ✅ 内存占用小（按需处理）
- ✅ 适合大数据集

**劣势**：

- ⚠️ 首次访问有延迟
- ⚠️ 需要管理缓存

### Q3: LoRA vs 全量微调的核心区别是什么？

**标准答案结构**：

**根本区别**：

- **全量微调**：训练所有参数 W → W + ΔW (参数量：100%)
- **LoRA微调**：冻结基础模型，训练低秩适配器 A×B (参数量：1-3%)

**数学原理**：

```
全量微调：
h = Wx + b  (W ∈ ℝ(d×k)，训练所有参数

LoRA微调：
h = Wx + b + BAx  (A ∈ ℝ(d×r), B ∈ ℝ(r×k), r << min(d,k))
只训练A和B矩阵，W冻结
```

**显存优势**：

```
7B模型全量微调：
- 模型参数：14GB (FP16)
- 梯度和优化器：12GB
- 总显存：26GB

7B模型LoRA微调：  
- 基础模型：14GB (冻结)
- LoRA参数：0.3GB (可训练)
- 梯度和优化器：0.6GB
- 总显存：8.7GB
```

**效果对比**：

- 简单任务：LoRA效果接近全量微调
- 复杂任务：LoRA可能损失2-3%效果
- 但LoRA训练成本低10倍，性价比更高

- 🎓 **适用**：8GB显存训练7B模型

**AdaLoRA**：

- 🎯 **目标**：自适应优化rank分配
- 📊 **效果提升**：比固定rank效果更好
- 🔧 **复杂度**：实现复杂，训练不稳定
- 🎓 **适用**：追求极致效果的场景

### Q5: DeepSpeed的CPU卸载是如何工作的？

**标准答案结构**：

**CPU卸载原理**：

```
训练过程中的内存使用：

GPU显存 (11GB):
├─ 模型参数: 8GB
├─ LoRA参数: 0.3GB  
└─ 激活值: 2.7GB (需要很大空间)

传统方法: 全部放在GPU → 显存不足
CPU卸载: 优化器和参数放到CPU → 节省GPU显存
```

**卸载策略**：

**优化器卸载** (第57-58行):

```bash
--offload_optimizer_device cpu
# AdamW优化器状态 (动量、方差) 存在CPU内存
# 节省: 0.5GB GPU显存
# 代价: CPU↔GPU通信延迟
```

**参数卸载** (第58行):

```bash
--offload_param_device cpu  
# 模型参数的反向传播梯度暂存CPU
# 节省: 0.3GB GPU显存  
# 代价: 需要额外的CPU内存
```

**通信流程**：

```
前向传播: GPU → GPU (快速)
梯度计算: GPU → CPU → GPU (稍慢)
参数更新: CPU → GPU (慢)
```

**效果分析**：

- 💾 显存节省: 0.8GB
- ⚡ 速度影响: 20-30%
- 🎯 适用: 单GPU训练最佳
